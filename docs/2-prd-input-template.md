## 7. API 설계 가이드 **[필수]**

### 7.1 API 엔드포인트 구조
**질문**: API 엔드포인트를 어떻게 설계하시겠습니까?

```

베이스 URL: http://localhost:3000/api

인증 방식:

- [x] JWT (Access Token + Refresh Token) - 도메인 정의서에서 결정됨
- [ ] Session-based
- [ ] OAuth 2.0

주요 엔드포인트 (예시):

[인증]
POST /api/auth/register - 회원가입
POST /api/auth/login - 로그인
POST /api/auth/logout - 로그아웃
POST /api/auth/refresh - 토큰 갱신

[할일]
GET /api/todos - 할일 목록 조회
POST /api/todos - 할일 생성
GET /api/todos/:id - 할일 상세 조회
PUT /api/todos/:id - 할일 수정
PATCH /api/todos/:id/complete - 할일 완료
DELETE /api/todos/:id - 할일 삭제 (휴지통)
PATCH /api/todos/:id/restore - 할일 복원

[휴지통]
GET /api/trash - 휴지통 조회
DELETE /api/trash/:id - 영구 삭제

[국경일]
GET /api/holidays - 국경일 조회
POST /api/holidays - 국경일 추가 (관리자)
PUT /api/holidays/:id - 국경일 수정 (관리자)

```

### 7.2 에러 코드 정의 (선택)
**질문**: 표준 에러 코드 체계를 정의하시겠습니까?

```

- 400: Bad Request (잘못된 요청)
- 401: Unauthorized (인증 실패)
- 403: Forbidden (권한 없음)
- 404: Not Found (리소스 없음)
- 409: Conflict (중복 데이터)
- 500: Internal Server Error (서버 오류)

- TODO_NOT_FOUND: 할일을 찾을 수 없음
- ALREADY_DELETED: 이미 삭제된 할일

```
